# WorkFlow AI 최종 기능 정리

## 1. 서비스 정의

WorkFlow AI는 대학생 팀프로젝트, 캡스톤디자인, 해커톤, AI 경진대회, 공모전 참가팀을 위한 AI 협업 플랫폼이다.

단순 업무 관리 도구가 아니라 회의록, 업무, 일정, GitHub, 산출물, AI 예측 모델, 심사자 평가 근거를 하나의 흐름으로 연결한다.

## 2. 핵심 차별점

| 구분 | 기존 협업툴 | WorkFlow AI |
| --- | --- | --- |
| 대상 | 일반 회사/조직 프로젝트 | 대학생 팀프로젝트, 해커톤, 경진대회, 공모전 |
| 회의록 | 기록/첨부 중심 | AI 요약, 할 일 생성, 역할 분배, 대시보드 반영 |
| 업무 관리 | 칸반/이슈 중심 | 카테고리별 상세 UI + 개인 할 일 자동 반영 |
| AI 활용 | 문서 요약 또는 Assistant 중심 | LLM/RAG + ML 예측 + DL 문장 분류/STT |
| 산출물 | 별도 작성 | 발표자료, 보고서, README, 제안서, 실험 보고서 자동 생성 |
| 평가 | 없음 또는 단순 활동 로그 | 심사자 전용 기여도 리포트, AI 평가 근거, 공개 설정 |

## 3. 사용자 역할

| 역할 | 핵심 목적 | 주요 기능 |
| --- | --- | --- |
| 팀장 | 프로젝트 운영과 업무 분배 | 업무 생성/배정, 대시보드 관리, 회의록 승인, 산출물 검수 |
| 팀원 | 본인 업무 수행과 피드백 확인 | 개인 할 일, 진행률 업데이트, 블로커 등록, 개인 코멘트 확인 |
| 심사자 | 팀별 평가와 기여도 근거 확인 | 개인별 기여도 리포트, AI 평가 근거, 최종 점수 입력/공개 |

### 4.2 업무 보드

업무 보드는 상태 기준으로 4개 컬럼을 가진다.

- 할 일
- 진행 중
- 보류/블로커
- 완료

업무 생성 시에는 상태보다 먼저 카테고리를 선택한다.

| 카테고리 | 예시 |
| --- | --- |
| 기획 | 요구사항, 기능 정의, 일정 계획 |
| 리서치 | 시장조사, 논문조사, 경쟁 서비스 분석 |
| UX/UI | 사용자 흐름, 와이어프레임, Figma |
| 디자인 | 로고, 포스터, 발표자료 디자인 |
| 프론트엔드 | 화면 구현, 컴포넌트, 반응형 UI |
| 백엔드 | API, 인증, 권한, 서버 로직 |
| AI/ML | 모델 학습, 추론, 추천 알고리즘 |
| 데이터 | 수집, 전처리, 라벨링, EDA |
| DB | ERD, 테이블, 쿼리 최적화 |
| DevOps | 배포, 서버, CI/CD |
| GitHub | 브랜치, 커밋, 코드리뷰 |
| QA/테스트 | 테스트 케이스, 버그 리포트 |
| 보안 | 인증/인가, 취약점 점검 |

| 산출물 | 최종 제출 파일 |
| 운영/제출 | 제출처, 제출 마감, 최종 검수 |

### 4.3 회의록 AI

| 업로드 유형 | 처리 방식 | 결과 |
| --- | --- | --- |
| 문서 업로드 | 텍스트 추출 후 LLM 분석 | 요약, 결정사항, 위험요소, To-Do |
| 음성파일 업로드 | Whisper STT 후 분석 | 회의록 텍스트, 요약, 역할 분배 |
| 영상파일 업로드 | FFmpeg 음성 추출 + STT | 화상회의 기록 분석 |

회의록 분석 결과는 다음 기능에 연결된다.

- 업무 보드 할 일 생성
- 개인 할 일 자동 반영
- 목표/마일스톤 진행률 반영
- 대시보드 전체 업무 수/완료율/위험 업무 반영
- AI Assistant의 프로젝트 맥락으로 저장
- 산출물 생성 시 참고 데이터로 활용

| 업무 우선순위 분류 | 업무 제목, 설명, 회의 근거 | 높음/중간/낮음 | 업무 생성 |

- 표준 공개 데이터셋 없음. 팀원별 업무 분배는 조직 내부 스키마 의존도가 너무 높아 학습 초기엔 직접 정의한 규칙 기반 점수(룰 베이스)로 시작 → 서비스 운영하며 라벨 축적하는 게 정석

## 6. 추천 최종 기술 스택

버전은 서로 호환이 검증된 조합으로 고정한다. 새 버전 도입 시 아래 "호환성 근거"를 함께 갱신한다.

### 6.1 Frontend

| 기술 | 버전 |
| --- | --- |
| Node.js | 20.x LTS (20.16 이상) |
| React | 18.3.1 |
| Vite | 5.4.x |
| TypeScript | 5.5.x |
| Tailwind CSS | 3.4.x |

호환성 근거: Vite 5는 Node 18 이상을 요구하므로 Node 20 LTS로 통일. React 18.3은 TypeScript 5.5의 `@types/react` 18.3 타입 정의와 매칭된다. Tailwind 3.4는 Vite 5 + PostCSS 8 조합에서 공식 지원.

### 6.2 Backend

| 기술 | 버전 |
| --- | --- |
| Java | 17 LTS |
| Spring Boot | 3.3.x |
| Spring Security | 6.3.x (Spring Boot 3.3 BOM에 포함) |
| Spring Data JPA | 3.3.x (Spring Boot 3.3 BOM에 포함) |
| JWT | jjwt 0.12.x |
| Build Tool | Gradle 8.8.x |

호환성 근거: Spring Boot 3.x는 Java 17 이상을 요구하며, Spring Security/Spring Data JPA 버전은 Spring Boot 3.3 BOM이 관리하므로 개별 버전을 직접 지정하지 않고 BOM을 따른다. jjwt 0.12는 Java 17과 호환.

### 6.3 AI Backend

| 기술 | 버전 |
| --- | --- |
| Python | 3.11.x |
| FastAPI | 0.111.x |
| Pydantic | 2.7.x (FastAPI 0.111 요구사항) |
| Uvicorn | 0.30.x |

호환성 근거: FastAPI 0.111은 Pydantic v2(2.7 이상)를 요구한다. PyTorch 2.3과 Transformers 4.4x 계열은 Python 3.11까지 공식 지원하며, 3.12는 일부 ML 패키지 wheel이 늦게 제공되므로 3.11로 고정.

### 6.4 Database / Vector DB

| 기술 | 버전 |
| --- | --- |
| PostgreSQL | 16.x |
| pgvector | 0.7.x (PostgreSQL 13 이상 요구) |
| Chroma | 0.5.x |
| FAISS | 1.8.x |

호환성 근거: MySQL과 PostgreSQL 중 하나만 선택하되, pgvector를 쓰려면 PostgreSQL 계열이 필요하므로 PostgreSQL 16으로 통일. pgvector 0.7은 PostgreSQL 13~16을 지원.

### 6.5 ML / DL

| 기술 | 버전 |
| --- | --- |
| NumPy | 1.26.x |
| scikit-learn | 1.5.x |
| XGBoost | 2.1.x |
| LightGBM | 4.4.x |
| PyTorch | 2.3.x (CPU 또는 CUDA 12.1 빌드) |
| Hugging Face Transformers | 4.42.x |
| Tokenizers | 0.19.x (Transformers 4.42 요구) |
| KoBERT/KLUE-BERT | Transformers 4.42 호환 체크포인트 사용 |
| Whisper | openai-whisper 20231117 또는 faster-whisper 1.0.x |

호환성 근거: PyTorch 2.3은 NumPy 2.0을 아직 완전 지원하지 않으므로 NumPy 1.26으로 고정해 scikit-learn/XGBoost/LightGBM과 충돌을 방지. Transformers 4.42는 Tokenizers 0.19, PyTorch 2.1~2.3을 요구.
