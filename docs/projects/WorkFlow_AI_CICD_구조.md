# WorkFlow AI CI/CD 구조

## 1. 개요

로컬 브랜치 작업 → `dev` 병합 → 검증 통과 시 `main` 이동 → OCI 배포 순서로 진행하는 3단계 브랜치 승격 파이프라인이다.

```
[개인 로컬 브랜치] → push/PR → [dev] → (검증 통과) → [main] → OCI 배포
```

## 2. 브랜치 전략

| 브랜치 | 역할 | 배포 대상 |
| --- | --- | --- |
| `feature/*` | 각자 담당 기능 개발 (기능 오너제 기준 FS-1~FS-7 등) | 없음 (로컬/개인) |
| `dev` | 통합 브랜치. 기능 브랜치들이 모이는 지점 | 개발/스테이징 환경 (선택) |
| `main` | 배포 브랜치. 항상 배포 가능한 상태 유지 | OCI 운영 환경 |

- 기능 개발은 항상 `feature/*` 브랜치에서 진행하고 `main`에 직접 push하지 않는다.
- `dev`도 가급적 PR을 통해 병합하며, 직접 push는 지양한다.

## 3. 단계별 흐름

### 3-1. 로컬 브랜치 작업

- 각자 `feature/<기능명>` 브랜치에서 작업
- 커밋 전 로컬에서 최소 검증(빌드/테스트/린트) 수행
- 작업 시작 전 `dev` 기준 최신화(`git pull` 또는 `rebase`)

### 3-2. `dev`로 푸시 및 병합

- `feature/*` → `dev` PR 생성
- PR 단계에서 CI 실행: 빌드, 단위 테스트, 린트/타입체크
- 리뷰 승인 + CI 통과 후 `dev`에 병합
- `dev`는 여러 기능이 합쳐지는 지점이므로 병합 직후 통합 테스트(있다면 E2E 포함)로 회귀 여부 확인

### 3-3. `main`으로 이동

- `dev`가 일정 기간 안정적으로 유지되고 문제가 없다고 판단되면 `dev` → `main` PR 생성
- `main`으로의 병합은 배포를 트리거하므로 다음을 충족한 뒤 진행:
  - `dev`에서의 CI(빌드/테스트) 전체 통과
  - 알려진 블로킹 이슈 없음
  - (선택) 수동 QA/스모크 테스트 통과
- `main` 병합은 신중하게, 가능하면 소수 인원 승인 후 진행

### 3-4. OCI 배포

- `main` 병합 시 CI/CD 파이프라인이 OCI(Oracle Cloud Infrastructure)로 자동 배포
- 배포 대상: FastAPI 백엔드, Spring Boot 백엔드, 프론트엔드(React/Vite), (필요 시) Postgres 등 인프라 구성요소
- 배포 후 헬스체크로 정상 기동 확인
- 배포 실패 또는 배포 후 이상 발견 시 롤백 절차 필요 (이전 `main` 커밋으로 되돌리기 또는 이전 배포 아티팩트 재배포)

## 4. 검증(CI) 항목 (제안)

| 단계 | 항목 |
| --- | --- |
| PR 생성 시 (feature → dev) | 빌드, 단위 테스트, 린트/타입체크 |
| dev 병합 후 | 통합 테스트, (있다면) E2E |
| dev → main PR | 전체 테스트 재확인, 블로킹 이슈 점검 |
| main 병합 후 | OCI 배포, 배포 후 헬스체크 |

## 5. 롤백 및 장애 대응

- 배포 직후 문제가 발견되면 `main`의 직전 정상 커밋으로 롤백하고 재배포
- 원인 파악은 `docs/trouble-shooting` 폴더에 기록 (본 프로젝트 CLAUDE.md 컨벤션)
- 반복되는 배포 실패 패턴이 있다면 CI 검증 항목에 반영해 재발 방지

## 6. 향후 보완 사항 (미확정)

- `dev`→`main` 승격 기준(예: 최소 대기 기간, 필수 승인자 수)의 명문화
- OCI 배포 자동화 파이프라인(GitHub Actions 등)의 구체적 워크플로 정의
- 스테이징 환경을 `dev` 브랜치에 실제로 연결할지 여부
- 배포 실패 시 자동 롤백 여부(수동 vs 자동)
