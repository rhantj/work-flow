# WorkFlow AI 기획서 작성 가이드

> 각 항목에 어떤 내용을 채워야 하는지 정리한 문서입니다.
> 실제 기획서 작성 시 이 가이드를 참고해 항목별로 내용을 채워나가면 됩니다.

---

## 1. 프로젝트 개요

### 1.1 프로젝트 주제
- 서비스 한 줄 정의 (예: "AI 기반 업무 자동화 협업 플랫폼")
- 프로젝트명(WorkFlow AI)의 의미와 네이밍 배경
- 어떤 문제를 어떤 방식으로 해결하는지 핵심 요약 2~3줄

### 1.2 추진 배경 및 문제 인식
- 시장/업무 환경의 변화 (원격·협업 툴 확산, 반복 업무 증가 등)
- 현재 사용자가 겪는 구체적 페인포인트 (수작업, 툴 분산, 정보 파편화)
- 문제를 뒷받침할 근거 (통계, 설문, 사례 등)
- "왜 지금 이 서비스가 필요한가"에 대한 논리

### 1.3 개발 목적
- 프로젝트로 달성하려는 목표 (정량/정성)
- 해결하려는 핵심 문제 3~5가지
- 성공 기준(KPI) 초안 (예: 업무 처리 시간 30% 단축)

### 1.4 주요 사용자
- 타깃 사용자 정의 (페르소나 2~3명: 직무·연령·업무 상황)
- 사용자 유형별 니즈 (관리자 / 팀원 / 게스트 등)
- 사용 시나리오 개요

### 1.5 서비스 기대 효과
- 사용자 관점 효과 (시간 절약, 협업 효율)
- 조직/비즈니스 관점 효과 (생산성, 비용 절감)
- 사회적/부가적 효과 (선택)

## 2. 서비스 분석 및 차별화 전략

### 2.1 유사 서비스 벤치마킹
- 경쟁/유사 서비스 3~5개 선정 (Notion, Slack, Asana, Trello 등)
- 서비스별 핵심 기능·강점·가격 비교표
- 벤치마킹 포인트 정리

### 2.2 기존 서비스의 한계
- 경쟁 서비스가 해결하지 못한 문제
- AI 활용 부족, 자동화 한계, 학습 곡선 등
- 사용자 불만 사례 정리

### 2.3 WorkFlow AI 서비스 방향
- 우리가 지향하는 서비스 방향성
- 기존 서비스와 다른 접근 방식
- 핵심 가치 제안 (Value Proposition)

### 2.4 핵심 차별화 포인트
- 경쟁 대비 명확한 차별점 3~5가지 (예: AI 자동 업무 분배, 자연어 워크플로우 생성)
- 각 차별점이 사용자에게 주는 이점

### 2.5 주요 기능 요약
- 핵심 기능 리스트 (기능명 + 한 줄 설명)
- 기능 우선순위 (MVP / 확장)
- 기능 맵 또는 요약표

---

## 3. 요구사항 및 정책 정의

### 3.1 기능 요구사항 정의
- 기능 목록을 표로 정리 (기능 ID / 기능명 / 설명 / 우선순위)
- 기능별 상세 동작 정의

### 3.2 사용자 권한별 요구사항
- 권한 등급 정의 (관리자 / 일반 사용자 / 게스트)
- 권한별 접근 가능 기능·데이터 매트릭스

### 3.3 비기능 요구사항
- 성능 (응답 속도, 동시 접속자 수)
- 보안 (인증, 암호화, 접근 제어)
- 확장성, 가용성, 유지보수성
- 호환성 (브라우저, 디바이스)

### 3.4 정책 정의서
- 회원가입/탈퇴 정책
- 데이터 보관·삭제 정책
- 요금/구독 정책 (있는 경우)
- 개인정보 처리 방침 개요

### 3.5 서비스 제약 조건
- 기술적 제약 (API 한도, 인프라)
- 일정/자원 제약
- 법적/규제 제약

## 4. 시스템 구조 및 기술 설계

### 4.1 전체 시스템 아키텍처
- 시스템 구성 다이어그램 (프론트 / 백엔드 / DB / AI / 외부 API)
- 각 계층 역할 설명

### 4.2 서비스 구성도
- 서비스 모듈 구성 (인증, 워크플로우, 알림 등)
- 모듈 간 상호작용

### 4.3 AI 처리 구조
- 사용 AI 모델/API (LLM, 임베딩 등)
- AI 요청·응답 처리 흐름
- 프롬프트 설계 개요, 후처리 로직

### 4.4 데이터 처리 구조
- 데이터 수집 → 저장 → 처리 → 제공 흐름
- 캐싱, 배치, 실시간 처리 여부

### 4.5 기술 스택 및 개발 환경
- 프론트엔드 / 백엔드 / DB / 인프라 스택
- 개발 도구, 버전, 협업 환경
- 선택 이유 간단 명시

### 4.6 외부 API 및 연동 구조
- 사용 외부 API 목록 (AI, 인증, 알림, 결제 등)
- 연동 방식과 인증 처리
- 장애 대응/폴백 전략

## 5. 서비스 흐름 및 화면 설계

### 5.1 메뉴 구조도 / 사이트맵
- 전체 화면 계층 구조 (트리 형태)
- 주요 메뉴와 하위 페이지

### 5.2 사용자별 서비스 흐름
- 사용자 유형별 이용 흐름 (관리자 / 팀원)
- 진입부터 목표 달성까지 경로

### 5.3 핵심 업무흐름도
- 주요 업무 프로세스 플로우차트
- 분기/예외 처리 포함

### 5.4 시스템 흐름도
- 요청 → 처리 → 응답의 시스템 관점 흐름
- 데이터/제어 흐름

### 5.5 화면 설계서 / 스토리보드
- 주요 화면 와이어프레임
- 화면별 구성 요소·동작 설명

### 5.6 주요 서비스 화면
- 핵심 화면 목업 (대시보드, 워크플로우 편집 등)
- 화면별 주요 인터랙션

### 6.2 주요 테이블 정의
- 테이블별 컬럼 정의서 (컬럼명 / 타입 / 제약 / 설명)

### 6.3 테이블 간 관계

- 참조 무결성·연관 규칙

- 핵심 기능별 데이터 CRUD 흐름
- 데이터 생명주기

## 7. 개발 수행 계획

### 7.1 팀원별 역할분담
- 팀원별 담당 영역·책임
- 역할 매트릭스

### 7.2 기능별 개발 오너 구성
- 기능별 주 담당자(오너) 지정
- 백업/리뷰어 지정

### 7.3 전원 풀스택 개발 방식
- 풀스택 방식 채택 이유
- 지식 공유·코드 이해 공유 방식

### 7.4 개발 일정계획
- 마일스톤·스프린트 계획
- 간트차트 또는 주차별 일정표

### 7.5 협업 및 코드 통합 방식
- Git 브랜치 전략 (main / feature 등)
- 코드 리뷰·머지 규칙
- 커뮤니케이션 채널

### 7.6 주요 산출물 계획
- 단계별 산출물 목록 (기획서, 설계서, 코드, 테스트 등)
- 산출물 제출 시점

### 8.2 트러블슈팅 계획

### 8.4 리스크 관리 방안
- 리스크 목록·우선순위 (기술/일정/인력)

### 8.5 향후 개선 방향
- 서비스 고도화 로드맵
