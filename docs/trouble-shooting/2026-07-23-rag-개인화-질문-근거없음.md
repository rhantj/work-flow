# RAG 개인화 질문이 "근거 없음"으로 답하던 문제

- 날짜: 2026-07-23
- 브랜치: `feature/ai_assistent`
- 관련 코드: `App/backend_fastapi/llm_rag_assistant/app/services/generation_service.py`

## 증상

"내 업무가 뭐야?" 같은 개인화 질문에 담당 업무가 실제로 있는데도
"근거 없음: 관련 자료를 찾지 못했습니다"가 반환됐다.

운영 DB(project 1, user 2 담당 task 18건)로 4회 측정한 결과 **0/4**.
간헐적 실패가 아니라 사실상 항상 실패하는 상태였다.

## 오해하기 쉬운 지점

검색이 잘못됐다고 보기 쉽지만 아니다. 검색은 정상이었다.

- `search_similar_chunks`가 `assignee_id` 필터로 담당 청크 5건을 정확히 반환
- 응답의 `sources` 배열에도 그 5건이 그대로 담겨 나감
- 즉 **출처는 보이는데 답변만 "근거 없음"** 이었다

로그에도 경고가 남지 않는다. 모델이 정상 응답한 결과가 거부 문구일 뿐이라
애플리케이션 입장에서는 성공이다.

## 원인

`_PERSONAL_CONTEXT_NOTICE`의 문구가 약했다.

```
아래 자료는 모두 질문자 본인이 담당자로 지정된 항목입니다.
```

청크 본문에는 업무 제목만 있고 담당자 이름이 없다. 모델은 위 안내문을 단정으로
받아들이지 않고 본문을 근거로 재판단했고, "컨텍스트만으로는 누구 업무인지
불명확하다"며 거부했다. 실제 거부 응답 문구:

> 제공된 컨텍스트에는 누구의 업무인지가 명확하지 않으며…

## 검증 (조합별 실측)

운영 DB 컨텍스트로 조합별 4회씩 측정했다.

| 조합 | "내 업무가 뭐야?" | "내 업무 마감일 알려줘" |
| --- | --- | --- |
| 현행 (temperature 미지정 + 기존 안내문) | 1/4 | 0/4 |
| temperature 0.1 + 기존 안내문 | **0/4** | 4/4 |
| temperature 0.1 + 강화 안내문 | **4/4** | **4/4** |
| temperature 미지정 + 강화 안내문 | 4/4 | 4/4 |

**temperature를 낮추는 것만으로는 고쳐지지 않는다.** 오히려 "누구 업무인지 모르겠다"는
잘못된 답에 일관되게 고정되어 1/4에서 0/4로 나빠졌다. temperature는 답을 고치지 않고
흔들림만 줄인다 — 진짜 수정은 안내문이다.

## 조치

`_PERSONAL_CONTEXT_NOTICE`를 다음으로 교체했다.

```
아래 자료는 시스템이 질문자 본인의 담당자 ID로 필터링해 가져온 것입니다.
따라서 아래 항목은 전부 질문자 본인의 담당 업무임이 이미 확정되어 있습니다.
본문에 담당자 이름이 없더라도 본인 업무로 간주하고 답하세요.
```

핵심은 두 가지다.

1. **필터링이 이미 끝났다**는 사실을 알린다 (모델이 재판단할 일이 아님)
2. **본문에 담당자 이름이 없어도 무방하다**고 명시한다 (거부 사유를 미리 차단)

함께 `_GENERATION_TEMPERATURE = 0.1`을 고정했다. 사실 조회형 응답이라 창의성이 필요 없고,
캐시된 답변과 재생성된 답변의 형식이 크게 달라지지 않는 편이 낫다.

## 반대 방향 회귀 확인

안내문을 강하게 만들면 근거가 없을 때도 지어낼 위험이 있다. 실제 API로 확인했다.

| 상황 | 결과 |
| --- | --- |
| 담당 업무가 없는 프로젝트(17)에 개인화 질문 | 근거 없음 (2/2) |
| 완전 무관한 질문("김치찌개 레시피") | 근거 없음 (2/2) |

안전장치는 유지된다.

## 최종 결과

실제 `/ai/rag/query` API로 각 3회 측정: **6/6 정상 응답**.

## 재발 방지

- 회귀 테스트 `test_personal_notice_states_ownership_is_already_confirmed`가
  안내문에서 "확정", "담당자 이름이 없더라도" 문구가 빠지는 것을 막는다.
- 프롬프트 구성이 바뀌면 `chat_service._ANSWER_CACHE_SCHEMA_VERSION`을 올려야 한다.
  이번 변경으로 `v3 → v4`. 올리지 않으면 배포 후 30분간 이전 프롬프트로 만든 답변이
  캐시에서 계속 반환된다.

## 남은 문제

프롬프트 품질은 단위 테스트로 검증할 수 없다. 테스트는 "문구가 들어 있는지"만 확인하며
"모델이 실제로 답하는지"는 확인하지 못한다. 프롬프트를 손댈 때는 반드시 실제 데이터로
여러 번 돌려 성공률을 세야 한다. 1회 실행 결과는 신호가 아니다 — 이번에도 같은 프롬프트가
연속 실행에서 성공과 실패를 오갔다.
