# 설계: delay_model.ipynb에 CatBoost / XGBoost / Random Forest 추가

## 배경

`App/backend_fastapi/ml_delay_risk/models/delay_model.ipynb`는 지연 위험도(정상/주의/위험)
3단계 분류를 LightGBM으로 학습하는 노트북이다. 상수 정의 → 피처 중요도 분석 → 피처 선정 →
이슈 단위 층화 분할 → SMOTE → 학습 → 평가 → 모델 저장 → 추론 테스트 순서로, 각 단계를
함수로 감싸지 않고 셀 단위로 직접 실행하며 중간 결과를 바로 확인하는 스타일이다.

이 설계는 같은 노트북에 CatBoost / XGBoost / Random Forest 3개 모델을 LightGBM과 동일한
파이프라인 스타일로 추가하고, 4개 모델을 검증 F1-Macro/precision/recall/f1-score/support로
비교하는 섹션을 만든다.

**피처 선정은 4개 모델이 공유한다** (아래 "공용 피처 선정" 참고 — 각 모델이 따로 피처
중요도를 계산하면 "어떤 모델이 더 유리한 피처를 골랐는가"가 알고리즘 자체 비교에 섞여
들어가므로, 상관계수(R²) 기반으로 한 번만 선정해 4개 모델 모두 같은 피처셋으로 비교한다).
이 때문에 기존 LightGBM의 피처 중요도/선정 셀(gain 기반 프로브 모델 학습)은 로직이
바뀐다 — 순수 추가만은 아니고, 이 부분만 재구성이 필요하다. 그 외 LightGBM 셀(분할 이후
SMOTE/학습/평가)은 그대로 둔다.

## 노트북은 완전히 순차 실행(stateful) 구조

셀들이 함수로 감싸여 있지 않고 전역 변수를 그대로 이어받는 구조이므로, 모델별 산출물은
접두사로 구분한다: LightGBM `lgm_`, CatBoost `cb_`, XGBoost `xgb_`, RandomForest `rf_`.
`df`/`candidate_features`/`train_df`/`test_df`/`selected_features`처럼 4개 모델이
동일하게 공유하는 값은 접두사 없이 그대로 둔다.

## 노트북 섹션 구성 (변경 후)

```
## 1. 환경 설정                      (변경 없음)
## 2. Imports                        (catboost/xgboost/RandomForest import 추가)
## 3. 상수 정의                      (변경 없음 — 데이터 관련 공유 상수)
## 3b. _RUN_TRAINING_CELLS           (변경 없음)
## 4. 모델 아티팩트 정의              (변경 없음 — delay_service.py가 그대로 참조)
## 5. 모델 저장/로드 함수             (변경 없음 — 함수 정의부, 위치 고정)
## 6. 실시간 추론용 함수              (변경 없음 — 함수 정의부, 위치 고정)
## 7. 학습 실행 — LightGBM, CatBoost, XGBoost, Random Forest
     ├─ 데이터 로드 ~ 이슈 단위 학습/검증 분할            (기존 셀, 거의 무변경)
     ├─ 공용 피처 선정 — 상관계수(R²) 기반                (로직 교체 — 아래 상세)
     ├─ 사용 피처 + 타겟 결합 데이터프레임                (기존 셀, selected_features로 표시)
     ├─ LightGBM: SMOTE → 학습 → 평가                    (기존 셀 그대로)
     ├─ CatBoost: 상수정의 → SMOTE → 학습 → 평가          (신규)
     ├─ XGBoost:  상수정의 → SMOTE → 학습 → 평가          (신규)
     └─ RandomForest: 상수정의 → 원-핫 → SMOTE → 학습 → 평가  (신규)
## 11. 4개 모델 성능 비교             (신규)
## 12. 모델 저장                     (기존 실행 셀 이동 — LightGBM artifact만)
## 13. 저장된 모델로 추론 테스트       (기존 실행 셀 이동 — LightGBM만)
```

`ModelArtifact`/`_save_artifact`/`load_artifact`/`predict_class_probabilities`
(4~6번, 함수 정의부)는 `delay_service.py`가 `_notebook_runtime.load()`로 그대로 가져다
쓰므로 이동하지 않는다. **이동 대상은 실행 셀**(아티팩트 생성+저장, 추론 테스트 실행)뿐이다.
모델 저장·추론 서빙은 계속 LightGBM(`booster`)만 대상으로 하고, 신규 3개 모델은 비교
실험용으로만 쓴다.

## 공용 피처 선정 — 상관계수(R²) 기반 (4개 모델 공용, 1회만 계산)

기존 LightGBM 전용 프로브 모델(gain 기반) 대신, 모델 없이 상관계수만으로 계산해서 4개
모델이 완전히 같은 `selected_features`를 쓰게 한다.

1. `freq_maps`(전체 데이터 기준 빈도 인코딩)는 기존과 동일하게 계산.
2. **수치형 + 빈도 인코딩 컬럼**: `encoded_df[col].corr(encoded_df["risk_class"])`로 피어슨
   상관계수 `r`을 바로 구하고 `r2 = r ** 2`.
3. **명목형 컬럼**(`CATEGORICAL_COLUMNS` — issuetype_name/priority_name/project_key/
   status_at_cutoff): 순서가 없는 범주라 임의의 정수 코드로 상관계수를 내면 코드 순서에
   따라 왜곡된다. 원-핫으로 펼쳐서 더미 컬럼별 상관계수를 구한 뒤, 같은 원본 컬럼에서 나온
   더미들의 `r2`를 합산해서 원본 컬럼 하나의 설명력으로 되돌린다.
4. 후보 피처 전체의 `r2`를 백분율로 정규화(`r2_pct`, 기존 `importance_gain_pct`와 같은
   개념)하고, 기존과 동일하게 1% 임계값으로 선정 → **`selected_features`** (4개 모델 공용,
   원본 `df` 컬럼 이름 리스트).
5. 막대그래프로 시각화 (`r2_pct` 기준, 기존 스타일 유지, `plt.show()`만).

이 결과가 "사용 피처 + 타겟 결합 데이터프레임" 단계(`display(df[selected_features +
["risk_class"]])`)와 이후 4개 모델 섹션 전체에서 그대로 재사용된다. 학습/검증 분할도
새로 나누지 않고 기존 `train_df`/`test_df`(이슈 단위 층화 분할)를 재사용한다.

## 모델별 반복 패턴 (SMOTE부터, 피처 선정은 이미 끝났으므로 생략)

1. **상수 정의**: 모델별 하이퍼파라미터 딕셔너리(`CATBOOST_PARAMS`/`XGBOOST_PARAMS`/
   `RF_PARAMS`)만 새로 정의.
2. **SMOTE 적용**: train 세트에만 적용.
3. **모델 정의 및 학습**: 조기 종료를 지원하는 모델(CatBoost/XGBoost)은 `test_df`를
   `eval_set`으로 넘겨 적용. 하이퍼파라미터는 튜닝 없이 LightGBM과 비슷한 스케일의
   베이스라인(`learning_rate=0.05`, `random_state=42` 등) — 이번 범위는 튜닝이 아니라
   베이스라인 비교.

### CatBoost / XGBoost — 범주형 처리

LightGBM과 동일하게 `CATEGORICAL_COLUMNS`를 pandas `category` dtype 그대로 사용.
CatBoost는 `cat_features` 인자로, XGBoost는 `enable_categorical=True` + `tree_method=
"hist"`로 네이티브 처리. SMOTE는 LightGBM과 동일하게 `SMOTENC`(범주형 컬럼 인덱스 지정).

### Random Forest — 범주형 처리 (원-핫 인코딩, 학습 입력 단계에서만)

피처 **선정**은 위 공용 단계에서 이미 원본 컬럼 단위로 끝났으므로, RF 섹션에서 할 일은
학습 입력을 만들 때만 원-핫으로 펼치는 것이다.

- `train_df`/`test_df`를 각각 `selected_features` 기준으로 잘라 원-핫 인코딩한 뒤,
  `test`를 `train`의 더미 컬럼 기준으로 `reindex(fill_value=0)`해서 정렬한다(두 세트에
  등장하는 카테고리 값이 다를 수 있으므로).
- **SMOTE는 `SMOTENC`가 아니라 일반 `SMOTE`를 쓴다**: 원-핫 인코딩 후에는 진짜 "범주형
  컬럼"이 없고 전부 numeric(0/1 포함)이다. `SMOTENC`로 더미 컬럼 각각을 독립적으로
  범주형 취급하면 원-핫의 "하나만 1" 제약이 깨질 수 있어(이웃마다 다른 더미가 1로 선택),
  일반 `SMOTE`로 전부 연속형처럼 보간하는 편이 단순하고 흔한 방식이다(보간된 더미 값이
  소수점이 되는 것은 알려진 근사이며 RandomForest 학습에는 문제 없음). 노트북에 이유를
  짧은 주석으로 남긴다.

## 4개 모델 비교 섹션 (## 11)

- 4개 모델(LightGBM 포함) 각각 자신의 인코딩된 `test_df`로 예측하고
  `classification_report(output_dict=True)`로 precision/recall/f1-score/support를
  뽑아 하나의 비교 표(모델 × 클래스)로 합친다.
- F1-Macro만 모은 막대그래프로 4개 모델을 한눈에 비교.
- 혼동행렬은 2×2 서브플롯 그리드로 4개 모델을 한 번에 시각화한다. `## 7`에 있던 LightGBM
  개별 혼동행렬 셀은 삭제하고(중복이므로) 여기 그리드로 통합한다.
- 그래프는 노트북 인라인 표시(`plt.show()`)만 하고 파일로 저장하지 않는다(기존 LightGBM
  셀과 동일 스타일).

## 의존성

`catboost`가 `requirements.txt`와 로컬 venv 어디에도 없다. `requirements.txt`에
버전 고정으로 추가하고 로컬 venv에 설치한다. xgboost(3.2.0)/scikit-learn(1.6.1)은
이미 설치돼 있다.

## 검증 계획

로컬 MongoDB(`ml_dashboard`)가 떠 있는 것을 확인했다. 노트북을 파일럿 사이즈
(`limit=1500`, 기존 기본값)로 실제 실행해서:
- 4개 모델이 에러 없이 학습되는지
- 비교 표/그래프/혼동행렬이 정상 출력되는지
- 기존 LightGBM 셀(저장/추론 테스트 포함, 이동 후 위치)이 여전히 정상 동작하는지

확인한다.
