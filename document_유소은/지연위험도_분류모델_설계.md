## 1. '업무 지연 위험도 분류(정상/주의/위험) 모델' 단계별 방법론

이진 분류(정상/지연)에서 한 단계 더 나아가 '정상/주의/위험'의 3단계 다중 분류(Multi-class Classification)로 모델을 고도화한다. 실무 대시보드에서 관리자가 개입할 타이밍을 결정하기에 실용적인 접근이다.

> **구현**: 실제로 `models/delayrisk_model.py`(코드 원본은 `delayrisk_model.ipynb` 노트북)가 LightGBM `objective="multiclass"`, `num_class=3`으로 정확히 이 3단계 분류를 구현했다.

### 1) 타겟 변수(Target) 3단계 정의 (Labeling)

과거 데이터에서 '정상(0)', '주의(1)', '위험(2)'을 수학적/논리적으로 명확하게 구분해 라벨링한다.

- **Class 0 (정상)**: 최초 산정된 마감일 이내에 원활하게 해결(Resolved)된 상태.
- **Class 1 (주의)**: 마감일은 지나지 않았으나, 특정 기한 내 진행률이 저조하거나 '블로커(Blocked)' 상태에 짧게(예: 전체 예상 시간의 10% 이하) 머물렀던 이슈.
- **Class 2 (위험)**: 마감일을 초과하여 지연되었거나, 장기간 '블로커' 상태에 빠져 활동이 완전히 멈춘(Silent) 이슈.

> **구현**: `models/feature_engineering.py`의 `classify_risk()` 함수가 이 정의를 그대로 규칙화했다.
>
> ```python
> if elapsed_ratio > 1.0:                     return 2  # 위험: 마감일(Proxy Deadline) 초과
> if blocked_ratio > risk_blocked_ratio(0.30): return 2  # 위험: 블로커 상태 장기 정체(Silent)
> if blocked_ratio > warning_blocked_ratio(0.10): return 1  # 주의: 블로커 상태 일정 기간 체류
> if imbalance_index > warning_imbalance_index(0.30): return 1  # 주의: 경과 대비 진행률 저조
> else:                                        return 0  # 정상
> ```
>
> 원본 문서가 예로 든 "전체 예상 시간의 10% 이하는 정상 범위"가 `warning_blocked_ratio=0.10`으로, "마감일 초과"가 `elapsed_ratio > 1.0`으로 정확히 대응된다. 이 임계값들은 `config.py`의 `Settings`에 상수로 빼두어 환경변수(`DELAY_RISK_*`)로 조정 가능하다.
>
> **다중 스냅샷 학습**: 설계 문서는 "과거 데이터에서 라벨링"이라고만 되어 있지만, 실제로는 완료된 이슈 하나를 **생성 후 1/3/7/14/30일 시점마다** 스냅샷으로 잘라 각각 라벨을 매긴다(`dataset_builder.py`). 즉 같은 이슈라도 초반 스냅샷은 '정상', 후반 스냅샷은 '위험'으로 서로 다른 라벨을 가질 수 있다 — "지금 이 순간의 상태"를 예측하는 모델의 목적에 맞춰, 학습 데이터도 시점별로 여러 번 샘플링한 것이다.

### 2) 다중 분류를 위한 핵심 피처 엔지니어링

티켓 생성일, 완료일, 담당자, 업무 예상량 등을 바탕으로 피처를 가공한다.

- **상태별 체류 시간 (Time in Status)**: 단순 현재 상태가 아닌, '대기'나 '블로커' 상태로 머문 누적 시간(예: 블로커 상태 누적 48시간).
  > **구현**: `feature_engineering.py`의 `_status_time_breakdown()`이 events의 상태 변경 changelog를 시간순으로 재생해 각 상태에 머문 시간을 전부 합산하고, `is_blocked_status()`(상태명에 `block`/`wait`/`hold`/`impediment` 포함 여부)로 판정된 시간만 골라 `blocked_hours_before_cutoff`로 반환한다.

- **진행률 불균형 지수 (Progress-Time Ratio)**: 단순 진행률뿐 아니라 '경과된 시간 대비 진행률'을 수치화. `timespent`(실제 투입 시간)와 `timeoriginalestimate`(최초 예상 시간)를 활용. 이 두 비율의 차이가 클수록(시간은 많이 지났는데 진행 안 됨) '주의'/'위험' 확률 ↑.
  > **구현**: `progress_ratio = time_spent_seconds / original_estimate_seconds`, `elapsed_ratio = elapsed_hours / proxy_deadline_hours`, `imbalance_index = elapsed_ratio - progress_ratio`로 정확히 구현. `timetracking` 객체의 초 단위(`originalEstimateSeconds`, `timeSpentSeconds`) 필드를 직접 나눠 문자열 파싱 없이 계산(핵심 피쳐 문서의 권고 그대로).
  >
  > **⚠️ 설계와 다르게 구현한 부분 — 데이터 누수(Leakage) 방지**: 설계 문서는 이 지수를 모델의 **피처**로 쓰라고 제안했지만, 실제로는 `elapsed_ratio_at_cutoff`, `blocked_ratio_at_cutoff`, `imbalance_index_at_cutoff` 세 값이 바로 위 `classify_risk()`의 **입력값 그 자체**다. 이걸 피처로 그대로 남겨두면 모델이 업무 패턴을 학습하는 게 아니라 라벨링 규칙식을 그대로 암기해버린다(타겟 누수). 그래서 노트북의 `LEAKY_FEATURE_COLUMNS`로 지정해 최종 학습 피처에서 **제외**했다. 대신 이 값들을 만드는 재료인 `progress_ratio_at_cutoff`, `elapsed_hours_at_cutoff`(절대 경과시간), `blocked_hours_before_cutoff`(절대 체류시간)는 피처로 남겨, 모델이 "비율"이 아니라 "원재료"로부터 스스로 패턴을 찾도록 했다. (처음 4개를 포함해 학습했을 때 검증 성능이 비정상적으로 높게 나온 것이 이 문제를 발견한 계기였다.)

- **마감일 잔여 시간 (Days until Due)**: 마감일까지 남은 시간을 연속형 변수로 변환.
  > **구현**: `hours_until_deadline_at_cutoff = proxy_deadline_hours - elapsed_hours_at_cutoff`로 계산은 하지만, 이 값 역시 `elapsed_ratio`를 역산할 수 있어(=사실상 동일 정보) 위와 같은 이유로 `LEAKY_FEATURE_COLUMNS`에 포함되어 **최종 피처에서는 제외**된다.

- **최근 활동 모멘텀 (Activity Logs)**: 하루 단위 댓글 수, 상태 변경 횟수 등. 기한 임박 + 최근 3일간 활동 0 → '위험'의 강력한 전조.
  > **구현**: `activity_count_recent_window`(기본 최근 3일, `config.py`의 `recent_activity_window_days`)가 댓글 + 상태/담당자 변경 이벤트 + 작업기록(worklog, 봇 제외) 건수를 합산한다. 다만 이 피처는 현재 `classify_risk()`의 라벨링 규칙에는 아직 반영되어 있지 않고(라벨은 elapsed/blocked/imbalance 3가지로만 결정), 모델 학습용 피처로만 제공된다 — 원본 설계가 제안한 "전조 신호"로서의 역할은 모델이 학습 과정에서 스스로 찾아내도록 맡긴 것.

### 3) LightGBM 모델링 전략 (Multi-class)

LightGBM은 범주형 데이터를 잘 처리하며 결측치에 강해 이 프로젝트에 최적.

- **Objective 설정**: `binary` 대신 `multiclass`로 변경, `num_class=3` 명시.
  > **구현**: `train_and_save()`의 `params = {"objective": "multiclass", "num_class": NUM_CLASSES, ...}` (`NUM_CLASSES = 3`). 그대로 구현.

- **Metric**: `multi_logloss`/`multi_error`로 오차 측정. '위험'을 '정상'으로 잘못 예측하는 게 더 치명적이므로 F1-Macro Score/혼동행렬로 클래스별 성능을 꼼꼼히 확인.
  > **구현**: 학습 중 모니터링은 `metric=["multi_logloss", "multi_error"]` + `early_stopping(30)`. 최종 평가는 `f1_score(..., average="macro")`, `classification_report`, 그리고 seaborn 히트맵으로 그린 혼동행렬(`models/confusion_matrix.png`)까지 전부 구현.

### 💡 현실적인 한계 및 극복 방안 (원본 Candor)

Zenodo 데이터셋 특성상 상당수 이슈에 명시적 '마감일(Due date)'이 없다. 우선순위나 스토리 포인트로 그룹화해 '과거 유사 티켓들의 평균 해결 시간'을 가상의 마감일(Proxy Deadline)로 대체하는 우회 전략이 필수적이다. 또한 '위험' 클래스가 '정상'보다 현저히 적은 클래스 불균형이 발생할 가능성이 높다.

> **구현 — Proxy Deadline**: `dataset_builder.py`의 `build_proxy_deadline_map()`이 `(issuetype, priority)` 조합별 과거 처리시간 **중앙값**을 계산한다. 원본이 언급한 "스토리 포인트"는 이 데이터셋에 없어 `issuetype`+`priority` 조합으로 대체했다. 표본이 `min_group_size_for_sla`(기본 10건) 미만인 희소 조합은 전역 중앙값으로 대체한다. 실제 실행 시: 그룹 30개, 전역 중앙값 521.9시간(표본 1,417건).
>
> **구현 — 클래스 불균형**: 실제로 우려대로 나타났다(정상 86.6% / 주의 0.8% / 위험 12.6%, 1,500건 파일럿 기준). 두 가지로 대응:
> 1. **역빈도 샘플 가중치**: `_sample_weights()`가 `total / (클래스 수 × 해당 클래스 표본 수)`로 계산한 가중치를 `lgb.Dataset(weight=...)`에 적용.
> 2. **StratifiedGroupKFold 분할**: 단순 랜덤 분할이면 희귀 클래스가 검증셋에 아예 없을 수 있어, 이슈(`issue_key`) 단위로 그룹을 통째로 배정하면서 그룹 대표 라벨(그 이슈가 도달한 최고 위험도)로 층화한다. (원본 문서에는 없는, 구현 과정에서 추가된 기법 — 아래 "설계에 없던 추가 사항" 참고)

## 2. '진행률 불균형 지수 (Progress-Time Ratio)' 상세

### 정의

단순 진행률뿐 아니라 '경과된 시간 대비 진행률'을 수치화한 지표. "주어진 시간 중 얼마를 썼는가?"와 "목표 업무량 중 얼마를 끝냈는가?"의 비율을 비교. 차이가 클수록(시간은 많이 지났는데 진행 안 됨) '주의'/'위험' 확률 ↑.

### 핵심 계산식

- **업무 진행률 (Progress Ratio)**: `timespent ÷ timeoriginalestimate`
- **시간 경과율 (Time Elapsed Ratio)**: `(현재 시점 - 생성일) ÷ (마감일 - 생성일)`
- **진행률 불균형 지수 (Imbalance)**: 위 두 비율의 차이

> **구현**: `feature_engineering.py`의 `build_dynamic_features()`에서
> ```python
> progress_ratio = time_spent_seconds / original_estimate_seconds   # None if 예상시간 없음
> elapsed_ratio  = elapsed_hours / proxy_deadline_hours              # 마감일 자리에 Proxy Deadline 사용
> imbalance_index = elapsed_ratio - progress_ratio                   # None-safe 처리
> ```
> `progress_ratio`가 `None`(예상시간 미설정)이면 `imbalance_index`도 `None`이 되고, LightGBM은 이를 결측치로 자연스럽게 처리한다(별도 대치 불필요 — LightGBM 선택 이유와도 연결됨).

### 지수 해석 가이드

| 불균형 지수 (Imbalance) | 상태 평가 | 현상 설명 | 예상 타겟 라벨 |
|---|---|---|---|
| 음수(−) 값 | 안전 | 경과된 시간보다 업무 진행이 더 빠름 (예: 시간 30% 경과, 진행 50%) | 정상 (Class 0) |
| 0에 근접 | 이상적 | 시간 흐름과 진행 속도가 거의 비례 | 정상 (Class 0) |
| 양수(+) 값(높음) | 경고/위험 | 시간은 많이 지났는데 진행 안 됨 (예: 80% 경과, 20% 진행) | 주의(1) 또는 위험(2) |

> **구현**: `classify_risk()`의 `imbalance_index > warning_imbalance_index(0.30)` 조건이 정확히 이 표의 "양수(높음) → 주의/위험" 판정을 담당한다. 다만 위에서 짚었듯, 이 지수 자체는 **라벨을 만드는 재료**라서 최종 학습 피처에서는 제외되고, 학습/검증 결과 리포트(F1-Macro, 혼동행렬)를 통해서만 "모델이 이 규칙을 잘 배웠는지"를 간접적으로 확인한다.

### 💡 실무 적용 시 주의점 (원본 AI의 조언)

IT/개발 업무 패턴은 선형적이지 않다(며칠간 진행률 0 → 마지막 날 급격히 100%). 이 지수를 마이크로매니징에 쓰지 말고, "과거 이 정도 불균형을 보인 이슈가 결과적으로 지연되었는가"를 통계적으로 분류하는 피처 중 하나로만 활용해야 한다.

> **구현 관점 코멘트**: 위에서 설명한 데이터 누수 방지 조치(비율 자체는 피처에서 제외, 원재료만 남김)가 결과적으로 이 조언과도 맞아떨어진다 — 모델이 "불균형 지수가 어느 값이면 위험"이라는 규칙을 직접 베끼는 대신, `time_spent_seconds_before_cutoff`, `elapsed_hours_at_cutoff` 같은 원재료와 다른 피처(블로커 이력, 활동 모멘텀 등)를 종합해 통계적으로 판단하게 된다.

## 3. 핵심 피쳐 가공 방법론

### 1) 새롭게 발견된 핵심 피처 가공 포인트

- **더욱 정교해진 '진행률 불균형 지수' 계산**: `progress` 객체 대신 `timetracking.originalEstimateSeconds`/`timeSpentSeconds`(초 단위 정수)를 직접 나눠 정확하고 빠르게 계산.
  > **구현**: 위 2번 섹션 참고 — 그대로 구현.

- **worklogs 내 봇(Bot) 데이터 필터링 필수**: GitHub 봇의 자동 PR 리뷰 코멘트 등을 `author` 필드에서 예외 처리해야 함.
  > **구현**: `models/bot_filter.py`의 `is_bot_author()` (`BOT_NAME_MARKERS = ("bot", "[bot]", "hudson", "jenkins", "buildbot", "github-actions")`). `build_dynamic_features()`가 진행률·활동량 계산 전에 항상 이 필터를 먼저 적용.

- **issuelinks를 활용한 연쇄 지연(Blocker) 파악**: 연결된 이슈의 status/priority 스냅샷으로, 나를 막고 있는 이슈가 아직 미해결인지 추적.
  > **구현**: `count_blocked_by_links()` → `num_blocked_by_links`, `num_unresolved_blockers`. 위 스키마 문서에서 언급했듯, 연결된 이슈의 `status`가 "현재 시점" 스냅샷이라는 한계는 남아있다.

### 2) 데이터 파이프라인 구축 시 치명적인 주의사항 (원본 Candor)

- **인덱스 부재로 인한 병목**: `events`(약 978만 건), `comments`(약 511만 건)에 `_id` 외 인덱스가 없어, `issue` 필드 기준 조회/조인 시 풀 스캔 발생 위험. MongoDB에서 PostgreSQL로 넘기기 전 `events`/`comments`/`worklogs`의 `issue`(또는 `issueId`)에 단일 인덱스 생성 필요.
  > **구현**: `models/mongo_client.py`의 `ensure_indexes()`가 정확히 이 세 컬렉션의 `issue`(및 `worklogs.issueId`)에 인덱스를 생성한다. `dataset_builder.py`가 학습 데이터 생성 전에 항상 먼저 호출.

- **불필요한 데이터 비대화(Bloat) 방지**: `issues.attachment[]`의 아바타 URL 등 불필요 메타데이터. PostgreSQL 적재 시 핵심 필드만 정제해 넣어야 함.
  > **구현 관점**: 이 프로젝트는 현재 PostgreSQL 이관 전이라(MongoDB 직접 조회), `attachment[]` 자체를 아예 쿼리/프로젝션하지 않는 방식으로 이 문제를 원천 차단했다. 참고로 `requirements.txt`에는 `# MongoDB ---> PostgreSQL로 이관하면 삭제하기`라는 코멘트와 함께 `pymongo`가 표시돼 있어, 팀에서 향후 PostgreSQL 이관을 계획 중임을 알 수 있다 — 이관 시 이 경고가 다시 유효해진다.

## 설계 문서에는 없던, 구현 과정에서 추가된 사항

원본 설계 문서 두 편에는 언급되지 않았지만, 실제 구현/실험 과정에서 필요해져 추가된 부분들:

1. **StratifiedGroupKFold 분할**: 단순 `train_test_split`이 아니라 `issue_key` 단위 그룹 분할 + 계층 층화. 같은 이슈의 여러 스냅샷이 train/valid에 걸쳐 들어가는 **그룹 누수**를 막고, 희귀 클래스('주의')를 검증셋에서도 보장한다.
2. **데이터 누수 피처 제거(`LEAKY_FEATURE_COLUMNS`)**: 위에서 설명한 대로, 설계 문서가 피처로 제안한 값들 중 일부가 실제로는 라벨 계산식 그 자체였음을 실험 중 발견하고 제거.
3. **다중 스냅샷 학습**: 완료 이슈 1건당 1개 라벨이 아니라, 생성 후 1/3/7/14/30일 시점마다 별도 학습 행을 생성. 실시간 추론(현재 시점 스냅샷)과 학습 데이터의 형태를 동일하게 맞추기 위함.
4. **train/serve 코드 공유**: `feature_engineering.py`의 피처 계산 함수와 인코딩 로직(`_apply_frequency_encoding`, `_apply_categorical_dtype`)을 학습(`train_and_save`)과 실시간 추론(`predict_class_probabilities`)이 그대로 재사용해 train/serve skew를 방지.
5. **혼동행렬 시각화**: 텍스트 로그 대신 seaborn 히트맵으로 렌더링, 한글 라벨 폰트(맑은 고딕) 설정 포함.
6. **모델 아티팩트에 부가 정보 동봉**: `booster` 외에 `feature_names`, `categorical_columns`, `frequency_maps`, `proxy_deadline_map`, `global_median_duration_hours`까지 함께 `joblib`으로 직렬화(`delayrisk_model.pkl`) — 추론 시 학습 때와 동일한 인코딩/Proxy Deadline 조회를 재현하기 위함.
