## 1. 배포할 때마다 20~40초 다운타임 (우선순위: 높음)

### 문제

`docker compose up -d --build`가 컨테이너를 교체하는 구조라, 컨테이너가 뜬 시점과
Spring이 요청을 받기 시작하는 시점 사이에 공백이 생긴다. 그 사이 모든 API가 502다.

### 근거

2026-07-23 PR #246 배포 중 실제 사용자 2명이 로그인 실패를 겪었다. nginx 로그:

```text
11:22:01  컨테이너 재생성
11:22:44  컨테이너 start (JVM 부팅 시작)
11:22:44~50  connect() failed (111: Connection refused) → 502 5건
          client: 221.153.94.253 (Windows), 1.220.247.75 (Mac)
11:23:05  Spring 기동 완료 (18.9초)
```

Spring 기동에만 19초가 걸리므로 이 공백은 구조적이다. 배포마다 반복된다.

### 선택지

| 안 | 내용 | 규모 | 상태 |
|---|---|---|---|
| A | 502/503/504에 "일시적, 재시도" 문구 부여 | 작음 | **완료** (`fix/api-unavailable-message`) |
| B | 로그인 등 핵심 요청에 자동 1회 재시도 | 중간 | 미정 |
| C | nginx가 백엔드 미기동 시 점검 안내 페이지 반환 | 중간 | 미정 |
| D | 무중단 배포 — 새 컨테이너가 health OK가 된 뒤 트래픽 전환 | 큼 | 미정 |

A는 오인을 없앨 뿐 다운타임 자체는 그대로다. **D가 근본 해결이다.**

### 권고

D를 하되 **단독 작업으로 분리한다.** 배포·롤백 경로는 2026-07-23에 이미 여러 번
고쳤고(3번 항목 참조) 종단 간 검증이 안 된 구간이 남아 있다. 같은 경로를 연달아
바꾸면 검증 부담이 겹친다. D 착수 전에 3번(스테이징)을 먼저 해결하는 편이 안전하다.

C는 D의 대체재가 아니라 보완재다. D를 하더라도 백엔드가 완전히 죽은 상황은 남는다.

## 2. 프론트엔드 CI가 없다 (우선순위: 높음)

### 문제

테스트 162개(28파일)가 레포에 있는데 **CI에서 아무도 실행하지 않는다.** 백엔드는
2026-07-23에 `backend-tests.yml`을 만들고 배포 게이트까지 연결했지만 프론트는 그대로다.

추가로 `App/frontend`에 **`tsconfig.json`이 없어** 타입체크 자체가 불가능하다.
`vite build`는 타입 오류를 잡지 않는다.

### 근거

```console
$ git ls-tree --name-only origin/dev .github/workflows/
backend-tests.yml
deploy-oci.yml
lint-workflows.yml

$ ls App/frontend/tsconfig*
(없음)
```

백엔드에서 정확히 이 공백 때문에 운영 장애가 났다(컨텍스트 기동 실패, `@Autowired`
누락). 프론트는 같은 공백이 그대로 남아 있다.

### 선택지

| 안 | 내용 | 규모 |
|---|---|---|
| A | `frontend-tests.yml` 추가 — PR/push에서 `vitest run` + `vite build` | 작음 |
| B | A + `tsconfig.json` 도입 후 `tsc --noEmit` 추가 | 중간 |
| C | B + `deploy-oci.yml`의 `test` 잡에 프론트 포함(배포 게이트) | 중간 |

### 권고

**A를 먼저 하고, B는 별도.** tsconfig 도입은 기존 코드에서 타입 오류가 대량으로
나올 가능성이 있어 규모를 예측하기 어렵다. 먼저 A로 회귀를 막고, B는 오류 개수를
확인한 뒤 결정한다. C는 B가 안정된 다음.

## 3. 롤백의 변경 구간이 검증된 적 없다 (우선순위: 높음)

### 문제

자동 롤백 스크립트에서 **판정부는 실측했지만 실제로 시스템을 바꾸는 구간은 한 번도
실행된 적이 없다.**

```text
검증됨   : 메트릭 수집(queue/outbox), 안전 가드, 롤백 대상 태그 선택
미검증   : docker stop → git reset --hard → docker compose up -d --build
```

### 근거

2026-07-23 조사에서 롤백 스텝이 **도입 이후 한 번도 동작한 적이 없었다**는 것이
드러났다(러너에서 bash 인용 오류로 exit 2, SSH 접속 전에 사망). 문법과 Go 템플릿
변수 문제를 고쳤고 판정부는 서버에서 읽기 전용으로 실측해 BLOCKED → PROCEED로
뒤집히는 것을 확인했다. 그러나 변경 구간은 운영에서 강제 실패를 유도할 수 없어
실행해 보지 못했다.

> 참고: `deploy-last-good` 태그 부재 문제는 PR #246 배포로 해소됐다.
> 현재 `deploy-last-good = de729d7`(정상 확인된 커밋)을 가리킨다.

### 선택지

| 안 | 내용 | 규모 |
|---|---|---|
| A | 스테이징 환경 구축 후 강제 실패를 유도해 종단 간 검증 | 큼 |
| B | 운영에서 저트래픽 시간대에 의도적 실패 배포로 1회 검증 | 중간 · 위험 |
| C | 롤백 스크립트를 dry-run 플래그로 분리해 변경 구간만 시뮬레이션 | 중간 |

### 권고

**A.** B는 실제 사용자에게 다운타임을 주고, 롤백이 실패하면 복구가 더 어려워진다.
C는 시뮬레이션일 뿐 실제 `git reset --hard`와 재빌드의 부작용을 재현하지 못한다.

스테이징은 1번(무중단 배포)에도 필요하므로 두 문제가 같은 선행 조건을 공유한다.

## 4. 마이그레이션 009/010의 SQL 파일이 레포에 없다 (우선순위: 중간)

### 문제

readiness 검사와 배포 preflight가 `meetings.analysis_job_id` 컬럼과
`rag_assignee_sync_failures` 테이블의 존재를 확인하는데, **이를 만드는 마이그레이션
파일이 레포에 없다.** Supabase에 수동 적용된 것으로 보인다.

신규 환경을 구축하면 스키마를 재현할 수 없고, readiness가 영구히 DOWN이 된다.

### 근거

```console
$ ls App/backend_spring/src/main/resources/db/migration/
V20260721_1__auth_and_project_onboarding.sql     # 이것 하나뿐
```

`deploy-oci.yml` preflight는 실패 시 "apply migrations 009 and 010 before
deployment"라고 안내하지만, 정작 그 009/010이 레포에 없다.

### 선택지

| 안 | 내용 | 규모 |
|---|---|---|
| A | 현재 운영 DB 스키마를 덤프해 마이그레이션 파일로 역작성 | 중간 |
| B | Flyway를 정식 도입하고 baseline을 현재 상태로 잡은 뒤 이후만 관리 | 중간 |

### 권고

**A 후 B.** 지금 스키마를 파일로 고정하지 않으면 시간이 갈수록 역작성이 어려워진다.
현재 `spring.flyway.enabled` 기본값이 `false`라 마이그레이션이 사실상 수동 운영이다.

## 5. 시크릿 로테이션 (우선순위: 중간)

### 문제

과거 채팅 기록에 평문으로 노출된 적이 있는 시크릿이 아직 교체되지 않았다.

- `RAG_INTERNAL_API_KEY` — Spring↔FastAPI 공유 시크릿. 양쪽이 같아야 한다.
- `LANGSMITH_API_KEY`

### 선택지

| 안 | 내용 | 규모 |
|---|---|---|
| A | 다음 배포 창에 맞춰 `.env` 교체 + 재기동 | 작음 |
| B | 시크릿 관리 도구 도입(Vault 등) 후 일괄 교체 | 큼 |

### 권고

**A.** `RAG_INTERNAL_API_KEY`는 양쪽이 동시에 바뀌어야 하므로 재기동이 필요하다.
1번(무중단 배포)이 해결되기 전이라면 다운타임을 감수할 시간대를 골라야 한다.
B는 지금 규모에 과하다.

## 6. 테스트 계정 동시 접속 제한이 시연 중 사고를 낼 수 있다 (우선순위: 중간)

### 문제

테스트 계정은 같은 계정으로 동시 로그인을 막는다. 해제는 두 경로뿐이다.

1. 로그아웃 버튼을 눌러 `/auth/test-logout` 호출
2. heartbeat가 끊긴 뒤 **TTL 40초** 경과

**브라우저 탭을 그냥 닫거나 새로고침하면 1번이 호출되지 않는다.** 그러면 40초 동안
그 계정으로 로그인할 수 없고, 사용자는 "다른 곳에서 로그인 중입니다"를 본다.

### 근거

`PresenceService.java:23` — `private static final Duration TTL = Duration.ofSeconds(40);`

2026-07-23 조사 중 이 상황을 여러 번 실제로 겪었다(`leader`, `member4`, `reviewer`).
중간보고·시연에서 발표자가 탭을 닫고 다시 열면 40초 대기가 발생한다.

### 선택지

| 안 | 내용 | 규모 |
|---|---|---|
| A | `beforeunload`/`pagehide`에서 `sendBeacon`으로 test-logout 전송 | 작음 |
| B | TTL을 10~15초로 단축 | 작음 |
| C | 같은 계정 재로그인은 기존 세션을 밀어내고 허용 | 중간 |

### 권고

**A + B.** A는 정상 종료 경로를 넓히고, B는 비정상 종료 시 대기 시간을 줄인다.
단 B만 하면 heartbeat 주기(20초)와 간격이 좁아져 오탐이 늘 수 있으므로 heartbeat
주기도 함께 조정해야 한다. C는 "동시 접속 제한"이라는 원래 의도를 없앤다 —
시연 중 두 사람이 같은 계정을 쓰는 상황을 막으려는 기능이므로 신중해야 한다.

## 7. 프론트엔드 번들이 단일 청크 1.9MB (우선순위: 낮음)

### 문제

```text
dist/assets/index-mZJyZCtP.js   1,888.94 kB │ gzip: 534.98 kB
(!) Some chunks are larger than 500 kB after minification.
```

gzip 후 535kB가 첫 진입에 전부 로드된다. 초기 로딩이 느리고, 배포마다 전체 번들이
무효화된다.

### 선택지

| 안 | 내용 | 규모 |
|---|---|---|
| A | 라우트 단위 `React.lazy` + `Suspense` 코드 분할 | 중간 |
| B | `manualChunks`로 vendor 분리 | 작음 |

### 권고

**B를 먼저.** 벤더 청크를 분리하면 배포 시 앱 코드만 무효화되어 재방문 로딩이
개선된다. A는 효과가 크지만 라우팅 구조를 건드려야 하므로 별도 작업.

우선순위는 낮다 — 현재 체감 문제로 보고된 적은 없다.
